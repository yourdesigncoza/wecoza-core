**quoteIdentifier enforcement (REPO-05):**

For MANUALLY-CONSTRUCTED SQL in repositories, add quoteIdentifier() to column names used in WHERE, SET, and ORDER BY clauses. Focus on the non-trivial manual SQL where column names are interpolated as strings (not in parameterized queries).

IMPORTANT: Do NOT add quoteIdentifier() to:
- Queries that already use parent methods (findBy, insert, update, delete) -- parent handles it
- Column names in SELECT lists (PostgreSQL handles these fine unquoted unless reserved)
- Column names that are hardcoded string literals in SQL (e.g., `WHERE status = 'deleted'`) -- these are safe
- Static methods using wecoza_db() directly -- different API, quoteIdentifier not available

Focus on instance methods that build dynamic WHERE/ORDER BY clauses with variable column names:
- `AgentRepository::getAgents()` -- the `$orderby` variable in ORDER BY clause. Wrap: `$this->quoteIdentifier($orderby)`
- `AgentRepository::getAgentNotes()` -- `$args['orderby']` in ORDER BY. Validate against allowlist first, then quote.
- `AgentRepository::getAgentAbsences()` -- `$args['orderby']` in ORDER BY. Validate then quote.
- `LearnerProgressionRepository::findWithFilters()` -- column names in conditions are hardcoded literals (safe), but ORDER BY uses hardcoded 'lpt.updated_at' (safe).

For most repositories, column names in SQL are hardcoded string literals -- these are already safe from SQL injection. The quoteIdentifier requirement mainly applies to cases where column names come from user input or variables.

Add a `// quoteIdentifier: column names are hardcoded literals (safe)` comment at the top of each repository's query methods section where applicable, to document that the audit was done.
